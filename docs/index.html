How it works

Open a project.
Ripple maps the risk.

Ripple scans supported JavaScript and TypeScript files locally, updates the graph as files change, and writes project-relative context that an AI agent can read before editing.

1

Ripple starts with a local scan of supported `.ts`, `.tsx`, `.js`, and `.jsx` files.
2

Import edges, caller counts, risk labels, and focus-file context are derived from the codebase in front of you.

Context files are generated locally
Project memory is written to .ripple/WORKFLOW.md, `.ripple/history.json`, and focused cache files for agents.

Risk classification

Agents know before
they touch anything.

Files are classified by blast radius from importers, churn, and usage signals. These labels are practical safety signals, not mathematical proof or a replacement for tests and code review.

● Dangerous
5+ importers or 15+ saves. Agent is instructed to stop and ask for explicit confirmation before changing shared code.
● Caution
2–4 importers or 8+ saves. Agent proceeds but verifies every caller still works after each change. Reports every file it touched.
● Safe
0–1 importers, low churn. Agent proceeds directly. Low blast radius means changes are usually more contained, but callers and tests still matter.

Honest scope

Ripple uses practical static analysis. It is a strong safety signal, not a mathematical proof. Namespace imports, aliased imports, dynamic imports, and some CommonJS patterns are part of the deeper semantic work planned after launch.
